    WebSocket & Real-Time Output in BackRunner

    How BackRunner streams program output to the browser over STOMP/WebSocket — endpoint, topics, SockJS fallback, and custom client connection examples.

    BackRunner streams all program output — compilation messages, runtime stdout, and error text — to the browser in real time over a STOMP-over-WebSocket connection. The built-in OUTPUT terminal panel in the IDE manages this connection for you automatically, but you can also subscribe from your own client code if you need to integrate BackRunner output into an external tool or dashboard.

    ​
    Protocol stack

    BackRunner uses the STOMP messaging protocol layered on top of a SockJS transport. SockJS provides automatic fallback to long-polling or HTTP streaming when a browser or network environment blocks native WebSocket upgrades, so the connection works reliably across restrictive corporate proxies.
    LayerTechnologyDetail
    TransportSockJSAutomatic WebSocket → HTTP fallback
    MessagingSTOMPPub/sub framing over SockJS
    Endpoint/ws-consoleRegistered on the BackRunner backend
    Output topic/topic/outputAll program output lines are published here
    App prefix/appUsed when sending messages to the backend
    The backend registers /ws-console twice: once as a raw WebSocket endpoint and once with SockJS enabled. You should always connect via the SockJS URL so that the fallback transport activates when needed.

    ​
    Connection details

    ​
    Endpoint URL

    http://<host>:8888/ws-console
    Replace <host> with the hostname or IP address of the machine running the BackRunner backend. When running locally the full URL is: 
    http://localhost:8888/ws-console

    ​
    Output subscription topic

    /topic/output
    Every line of output produced by a running program is published to this topic as a plain-text STOMP message. Each message body (msg.body) is a single string — a line of stdout, stderr, or a compiler diagnostic. Messages arrive in order and are not batched.

    ​
    Application destination prefix

    /app
    Use the /app prefix when you send a message to the backend (for example, triggering a run). You do not need this prefix when subscribing to /topic/output.

    ​
    Connecting from a custom client

    You need two browser libraries: SockJS and STOMP.js. Include them from a CDN, or bundle them with your build tool.
    • CDN (script tags)
    • npm
    <script src="https://cdnjs.cloudflare.com/ajax/libs/sockjs-client/1.6.1/sockjs.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/stomp.js/2.3.3/stomp.min.js"></script>

    npm install sockjs-client @stomp/stompjs

    ​
    Full JavaScript example

    const WS_URL         = 'http://localhost:8888/ws-console';
const SUBSCRIBE_TOPIC = '/topic/output';

// 1. Create a SockJS transport (handles WebSocket + HTTP fallback)
const sock   = new SockJS(WS_URL);

// 2. Wrap it in a STOMP client
const client = Stomp.over(sock);

// 3. (Optional) Suppress debug chatter in the browser console
client.debug = null;

// 4. Connect — the callback fires once the STOMP handshake completes
client.connect({}, function onConnected() {
  console.log('Connected to BackRunner');

  // 5. Subscribe to the output topic
  client.subscribe(SUBSCRIBE_TOPIC, function onMessage(msg) {
    // msg.body is a plain-text line from stdout / stderr / compiler
    console.log('OUTPUT:', msg.body);
  });

}, function onError(err) {
  console.error('Connection failed:', err);
});
    Stomp.over() accepts any SockJS instance. Passing the raw sock object — rather than a URL string — ensures STOMP uses the SockJS transport layer and benefits from automatic HTTP fallback.

    ​
    Disconnecting cleanly

    // Call disconnect when you no longer need the connection
client.disconnect(function () {
  console.log('Disconnected from BackRunner');
});

    ​
    Built-in terminal panel

    The IDE’s OUTPUT panel connects to the same /ws-console endpoint using the exact same SockJS + STOMP pattern. Understanding how it works helps you diagnose connection issues.
    1

    Libraries are loaded on demand

    When you click Connect in the OUTPUT panel, BackRunner dynamically injects the SockJS and STOMP script tags into the page if they are not already present. This keeps the initial page load lightweight.
    2

    SockJS connection is established

    A new SockJS instance is created pointing at http://localhost:8888/ws-console. SockJS negotiates the best available transport with the server.
    3

    STOMP handshake completes

    Stomp.over(sock).connect() performs the STOMP protocol handshake. The status indicator in the terminal title bar turns green once the handshake succeeds.
    4

    Subscription is registered

    The client subscribes to /topic/output. From this point every output line the server publishes appears as a new row in the OUTPUT panel in real time.

    ​
    Terminal status indicators

    Indicator colourMeaning
    🔴 RedDisconnected — click Connect to start
    🟡 AmberConnecting — STOMP handshake in progress
    🟢 GreenConnected — output is streaming

    ​
    Message format

    Each STOMP message delivered to /topic/output carries a plain-text string in its body. There is no JSON envelope or structured metadata — just the raw line of text. 
    msg.body  →  "Hello, World!\n"
msg.body  →  "error: ';' expected\n"
msg.body  →  "Process exited with code 0\n"
    Because messages are plain text, you can display them directly in a <pre> element or append them line-by-line to any terminal emulator. The OUTPUT panel uses white-space: pre-wrap so that indentation and newlines in compiler diagnostics are preserved exactly.

    ​
    CORS and allowed origins

    The backend WebSocket endpoint is configured with setAllowedOriginPatterns("*"), which means any origin is permitted to connect. You do not need to add custom CORS headers or configure an allow-list when connecting from a local development environment or a deployed frontend on a different domain.
    The wildcard origin pattern is suitable for development. If you expose BackRunner publicly, consider tightening setAllowedOriginPatterns on the backend to restrict WebSocket connections to your own domain.
