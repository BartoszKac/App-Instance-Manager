    Get Started

    Self-Host BackRunner with Docker or Maven

    Run BackRunner on your own machine or server. Spin up the SSH target containers with Docker Compose, or start the Spring Boot backend directly with Maven.

    BackRunner consists of a Spring Boot backend and a React frontend that are packaged together into a single runnable JAR. The backend serves the compiled frontend from its static resources, so you only need to start one process to get the full application. This page covers the two ways to run BackRunner: using Docker Compose to provision a ready-made set of SSH-accessible target containers alongside the application, and running the backend directly from source with Maven when you want a lighter setup or want to do development work.

    ​
    Prerequisites

    Before you begin, make sure the following tools are installed on the host machine.
    ToolMinimum VersionRequired For
    Java JDK17Maven build and running the JAR
    Maven3.6Building from source
    Docker24Docker Compose method
    Docker Composev2Docker Compose method
    g++Any modern releaseCompiling C++ code at runtime
    Python 33.8+Running Python scripts at runtime
    The embedded H2 database is bundled inside the JAR and writes its files to a ./data/ directory relative to where you start the server. You do not need to install or configure any external database. BackRunner creates the data/ folder automatically on first start.

    ​
    Method 1 — Docker Compose (Recommended)

    The repository ships with a docker-compose.yml and a Dockerfile that define five lightweight Alpine Linux containers. Each container runs an OpenSSH daemon and has Java 17, Python 3, and Bash pre-installed. These containers act as the remote deployment targets you manage from BackRunner’s Deploy and Instances views.

    ​
    Step 1 — Clone the repository

    git clone https://github.com/BartoszKac/App-Instance-Manager.git
cd App-Instance-Manager

    ​
    Step 2 — Review the compose file

    The docker-compose.yml in the repository root defines five target nodes:
    docker-compose.yml
    services:

  linux-1:
    build: .
    container_name: linux-1
    hostname: linux-1
    restart: unless-stopped
    ports:
      - "2221:22"
    volumes:
      - ./shared:/shared
    networks:
      - testnet

  linux-2:
    build: .
    container_name: linux-2
    hostname: linux-2
    restart: unless-stopped
    ports:
      - "2222:22"
    volumes:
      - ./shared:/shared
    networks:
      - testnet

  linux-3:
    build: .
    container_name: linux-3
    hostname: linux-3
    restart: unless-stopped
    ports:
      - "2223:22"
    volumes:
      - ./shared:/shared
    networks:
      - testnet

  linux-4:
    build: .
    container_name: linux-4
    hostname: linux-4
    restart: unless-stopped
    ports:
      - "2224:22"
    volumes:
      - ./shared:/shared
    networks:
      - testnet

  linux-5:
    build: .
    container_name: linux-5
    hostname: linux-5
    restart: unless-stopped
    ports:
      - "2225:22"
    volumes:
      - ./shared:/shared
    networks:
      - testnet

networks:
  testnet:
    driver: bridge
    Each container maps a host port (2221–2225) to SSH port 22 inside the container, so you can reach any of them from BackRunner’s Deploy view using localhost as the hostname. All five containers share a ./shared volume that maps to /shared inside each container. The Dockerfile that backs these containers is:
    Dockerfile
    FROM alpine:3.19

RUN apk add --no-cache \
    openssh \
    openjdk17-jre-headless \
    python3 \
    py3-pip \
    bash \
    curl

RUN ssh-keygen -A

RUN adduser -D -s /bin/bash user && \
    echo "user:user" | chpasswd

RUN sed -i 's/#PermitRootLogin prohibit-password/PermitRootLogin no/' /etc/ssh/sshd_config && \
    sed -i 's/#PasswordAuthentication yes/PasswordAuthentication yes/' /etc/ssh/sshd_config && \
    sed -i 's/#PubkeyAuthentication yes/PubkeyAuthentication yes/' /etc/ssh/sshd_config

EXPOSE 22

CMD ["/usr/sbin/sshd", "-D"]
    A non-root user named user with password user is created in each container. Use these credentials when registering servers in BackRunner’s Deploy view.

    ​
    Step 3 — Start the target containers

    docker compose up -d
    Docker builds the Alpine image once and starts all five containers in the background. Verify they are running: 
    docker compose ps

    ​
    Step 4 — Build and run the Spring Boot backend

    Navigate into the backend module and build the JAR: 
    cd App-Instance-Manager
mvn clean package -DskipTests
    Once the build succeeds, start the application: 
    java -jar target/DynamicCode-0.0.1-SNAPSHOT.jar
    The server starts on port 8888. Open http://localhost:8888 in your browser.

    ​
    Connecting to the Docker containers from BackRunner

    In BackRunner’s Deploy view, register a new server with the following credentials:
    FieldValue
    Hostnamelocalhost
    Port2221 – 2225 (one per container)
    Usernameuser
    Passworduser

    Method 2 — Maven Only (No Docker)

    If you only want to write and run code without the SSH deployment feature, or if you are working on the backend source, you can start the application directly without Docker.

    ​
    Step 1 — Start the backend

    cd App-Instance-Manager
mvn spring-boot:run
    Maven downloads all dependencies, compiles the project, and starts the embedded Tomcat server. The application is ready when you see a log line similar to: 
    Started DynamicCodeApplication in 3.4 seconds (process running for 3.9)
    Open http://localhost:8888 in your browser.

    ​
    Step 2 — (Optional) Start the frontend in development mode

    The Maven build embeds the compiled frontend inside the JAR, so the frontend is served automatically from http://localhost:8888 in production builds. If you want hot-reload during frontend development, run Vite separately: 
    cd App-Instance-Menager-fronted
npm install
npm run dev
    Vite starts a development server (typically on port 5173) that proxies API calls to the backend on port 8888.

    ​
    Configuration Reference

    All backend configuration lives in App-Instance-Manager/src/main/resources/application.properties. The most relevant options are listed below.
    application.properties
    # HTTP port the server listens on
server.port=8888

# H2 embedded database — file stored in the ./data/ directory relative to the working directory
spring.datasource.url=jdbc:h2:file:./data/backrunner;DB_CLOSE_ON_EXIT=FALSE
spring.datasource.driverClassName=org.h2.Driver
spring.datasource.username=sa
spring.datasource.password=

# Schema management — 'update' adds new columns without dropping existing data
spring.jpa.hibernate.ddl-auto=update

# Maximum size for a single uploaded file
spring.servlet.multipart.max-file-size=150MB

# Maximum size for the full HTTP multipart request
spring.servlet.multipart.max-request-size=150MB

# Embedded Tomcat form post size limit
server.tomcat.max-http-form-post-size=150MB
    You can override any property at runtime by passing it as a command-line argument: 
    java -jar target/DynamicCode-0.0.1-SNAPSHOT.jar --server.port=9000
    Or by setting environment variables using Spring Boot’s relaxed binding convention: 
    SERVER_PORT=9000 java -jar target/DynamicCode-0.0.1-SNAPSHOT.jar

    ​
    WebSocket Endpoint

    The frontend connects to the backend WebSocket endpoint at /ws-console using SockJS and the STOMP sub-protocol. Output from running programs is published to the /topic/output topic. If you are running BackRunner behind a reverse proxy, make sure it is configured to forward WebSocket upgrade requests. 
    ws://localhost:8888/ws-console
STOMP topic: /topic/output

    ​
    Production Considerations

    The default configuration is intended for local development and testing. Before exposing BackRunner on the public internet or a shared network, review the following points.
    • CORS is open by default. The backend allows requests from any origin. Restrict this to your actual frontend origin before exposing BackRunner on a shared or public network.
    • WebSocket allows all origin patterns. The WebSocket endpoint accepts connections from any origin by default. Tighten this to your domain before going live.
    • H2 database credentials. The default database username is sa with a blank password. If the H2 console is enabled, restrict or disable it.
    • SSH credentials in Docker containers. The bundled containers use user:user. These are suitable only for local sandboxes. Use strong credentials and key-based authentication for any internet-facing server.
    • File upload limits. The 150 MB limit covers large pre-built executables. Lower this if your environment does not require large uploads, to reduce the risk of abuse.
    • Reverse proxy. When running behind Nginx or a similar proxy, configure it to pass the X-Forwarded-* headers and to handle WebSocket upgrade requests on the /ws-console path.
