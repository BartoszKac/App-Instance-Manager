Every file you deploy in BackRunner is sent using one of three transfer strategies. The strategy tells the backend which form of the code to package and copy to the remote server over SSH. Picking the right strategy ensures the remote machine receives exactly the artifact it needs to run your program — no recompilation step required on the server if you ship bytecode or a binary, and no compiled output required if the server has its own runtime.

## The three strategies

<CardGroup cols={3}>
  <Card title="SOURCE_CODE" icon="code" color="#4f7ef7">
    Transfers the raw source files for the project — for example `.java`, `.py`, or `.cpp` files. Use this when the remote machine has a compiler or interpreter installed and you want it to build or run the code directly.
  </Card>
  <Card title="COMPILED_CODE" icon="microchip" color="#f59e0b">
    Transfers the compiled output — `.class` bytecode for Java or the compiled binary for C++. Use this when you have already compiled the code in BackRunner and the remote machine only needs the runtime (JVM, etc.), not a compiler.
  </Card>
  <Card title="REMOTE_EXECUTABLE" icon="circle-play" color="#22c55e">
    Transfers a pre-uploaded executable binary that was uploaded to BackRunner separately via the Upload screen. Use this when you have an existing binary and no source code is available, or when you want to keep your source code private.
  </Card>
</CardGroup>

## When to use each strategy

<Tabs>
  <Tab title="SOURCE_CODE">
    **Best for:** interpreted languages like Python and Bash, or any situation where the remote server handles compilation itself.

    - The remote machine must have the appropriate interpreter or compiler installed (e.g. `python3`, `g++`, `bash`).
    - BackRunner copies the raw `.py`, `.java`, `.cpp`, or `.sh` source files to the target directory on the remote server.
    - Default strategy for **Python** and **Bash** projects in BackRunner.
    - Useful when you want the remote environment to control the compiler version or build flags.

    ```
    Source project in BackRunner  →  SSH transfer  →  /home/deploy/project/Main.py
    ```
  </Tab>
  <Tab title="COMPILED_CODE">
    **Best for:** Java projects and C++ programs you have already compiled inside BackRunner.

    - The remote machine only needs the runtime, not a full compiler. For Java that means a JRE; for C++ binaries there are no additional dependencies beyond shared libraries.
    - BackRunner copies the compiled `.class` files (Java) or binary (C++) to the remote server.
    - Default strategy for **Java** projects in BackRunner.
    - Faster to start on the remote side because no compilation step is needed there.

    ```
    Compiled output in BackRunner  →  SSH transfer  →  /home/deploy/project/Main.class
    ```
  </Tab>
  <Tab title="REMOTE_EXECUTABLE">
    **Best for:** pre-built binaries you have uploaded via the Upload screen — no source code required.

    - The file was uploaded as a remote executable through `POST /api/process/upload` and is stored in BackRunner's executable library.
    - BackRunner retrieves the stored binary and transfers it to the remote server over SSH.
    - Useful for distributing compiled programs built outside BackRunner, or for shipping closed-source binaries.
    - The remote machine must be compatible with the binary architecture and OS.

    ```
    Stored executable in BackRunner  →  SSH transfer  →  /home/deploy/executables/myapp
    ```
  </Tab>
</Tabs>

## Choosing a strategy in the Deploy view

In the Files panel (left column of the Deploy view), every file row has a strategy dropdown immediately to the right of the file name. BackRunner pre-selects a default strategy based on the project's language, but you can override it independently for each file:

1. Click the strategy dropdown on the row of the file you want to change.
2. Select **Source**, **Compiled**, or **Executable** from the list.
3. The center panel's deployment plan updates immediately to reflect your choice.

<Info>
  You can use different strategies for different files in the same deployment. For example, you can ship a Java project as `COMPILED_CODE` and a Python script as `SOURCE_CODE` in a single click of the Deploy button.
</Info>

## API reference

### Single transfer

Use `POST /api/deploy/transfer/single` to deploy one project to one server with a specific strategy. Pass all three parameters as query parameters.

<ParamField query="mainClassId" type="long" required>
  The ID of the source project's main class entry.
</ParamField>

<ParamField query="idConfiguration" type="long" required>
  The ID of the target server's configuration (from `GET /api/deploy/servers`).
</ParamField>

<ParamField query="strategy" type="string" required>
  One of `SOURCE_CODE`, `COMPILED_CODE`, or `REMOTE_EXECUTABLE`.
</ParamField>

<Tabs>
  <Tab title="SOURCE_CODE">
    ```http
    POST /api/deploy/transfer/single?mainClassId=3&idConfiguration=7&strategy=SOURCE_CODE
    ```
    Transfers the raw source files for project `3` to server `7`.
  </Tab>
  <Tab title="COMPILED_CODE">
    ```http
    POST /api/deploy/transfer/single?mainClassId=3&idConfiguration=7&strategy=COMPILED_CODE
    ```
    Transfers the compiled bytecode/binary for project `3` to server `7`.
  </Tab>
  <Tab title="REMOTE_EXECUTABLE">
    ```http
    POST /api/deploy/transfer/single?mainClassId=12&idConfiguration=7&strategy=REMOTE_EXECUTABLE
    ```
    Retrieves the stored remote executable with ID `12` and transfers it to server `7`.
  </Tab>
</Tabs>

Both success and error cases return no body; check the HTTP status code (`200 OK` for success).

### Bulk transfer

Use `POST /api/deploy/transfer/bulk` to deploy multiple projects to multiple servers in one request. This is the endpoint the Deploy view uses when you click the **Deploy** button.

Send a JSON array of `TransferTask` objects in the request body:

<ParamField body="mainClassId" type="long" required>
  The ID of the project's main class entry.
</ParamField>

<ParamField body="configurationId" type="long" required>
  The ID of the target server's configuration.
</ParamField>

<ParamField body="uploadStrategyType" type="string" required>
  One of `SOURCE_CODE`, `COMPILED_CODE`, or `REMOTE_EXECUTABLE`.
</ParamField>

```http
POST /api/deploy/transfer/bulk
Content-Type: application/json
```

```json
[
  {
    "mainClassId": 3,
    "configurationId": 7,
    "uploadStrategyType": "COMPILED_CODE"
  },
  {
    "mainClassId": 5,
    "configurationId": 7,
    "uploadStrategyType": "SOURCE_CODE"
  },
  {
    "mainClassId": 3,
    "configurationId": 9,
    "uploadStrategyType": "COMPILED_CODE"
  }
]
```

The example above deploys project `3` as compiled code and project `5` as source code to server `7`, and also deploys project `3` as compiled code to a second server `9` — all in a single request. BackRunner returns `200 OK` with no body when all transfers succeed.

<Warning>
  If any transfer in a bulk request fails (for example, because SSH authentication is rejected), the backend logs the error for that specific task. Other tasks in the same batch may still succeed. Always check the deployment log in the center panel or your server logs for individual task outcomes.
</Warning>

## Strategy default by language

| Language | Default strategy | Rationale |
|----------|-----------------|-----------|
| Java | `COMPILED_CODE` | The JVM on the remote machine runs `.class` files directly; no compiler needed there. |
| Python | `SOURCE_CODE` | Python is interpreted; the remote machine runs `.py` files with its local `python3`. |
| C++ | `SOURCE_CODE` | Defaults to source; switch to `COMPILED_CODE` after compiling in BackRunner if the remote machine lacks a compiler. |
| Bash | `SOURCE_CODE` | Shell scripts are always transferred as-is and executed by the remote shell. |
